## 1. 文档目标与范围

本文档为「拾笺 NoteGather」个人知识库平台**第一阶段（Phase 1 · 核心 MVP）**的技术架构设计与落地方案，用于指导后续编码工作有序推进。

### 1.1 第一阶段目标

交付一个**基础可用**的知识库平台闭环：用户可以注册登录、创建与组织知识库/笔记、上传文档并异步解析入库、基于个人知识库进行 RAG 问答（带来源引用）。覆盖 Web 端与 Android 端基础功能。

### 1.2 第一阶段范围界定

**纳入范围（In Scope）**

- 用户注册 / 登录（用户名密码 + JWT 双 Token）
- 知识库 / 文件夹 / 笔记的 CRUD 与层级组织、拖拽排序、软删除回收站
- Markdown 笔记编辑（基础块编辑器）
- 文件上传 + 异步解析管线（文本抽取 → 分块 → 向量化 → 入库）
- 基础 RAG 问答（向量检索 + LLM 回答 + 来源引用，SSE 流式输出）
- Web 前端基础功能
- Android 端（Java + XML 原生）基础功能
- 本地 Docker Compose 开发环境

**排除范围（Out of Scope，后续阶段）**

- 第三方登录（微信 / Google 等集成）—— 本阶段不接入、不预留
- GraphRAG 知识图谱与可视化（Phase 2）
- 记忆卡与间隔重复（Phase 2）
- 陪伴模式 / 协作模式 / 实时协同（Phase 3）
- Agent 工作流引擎（Phase 3）
- iOS 端、浏览器插件、社区分享（Phase 4/5）

### 3.1 分层原则

- **技术栈隔离**：Java（`note-gather-server`）、Python（`note-gather-ai`）、Web、Android 各自独立工程，独立构建，互不侵入。
- **契约先行**：跨语言通信的 gRPC proto 统一放在根级 `protos/`，Java 与 Python 均从同一份 proto 生成代码，保证契约一致。
- **部署解耦**：中间件（DB/MQ/向量库等）与应用分离编排，开发期中间件用 Docker Compose 一键拉起。

### 4.2 分层规范（每个业务模块内部）

```
module-xxx/
├── controller/     # REST 接口层（参数校验、SSE 端点）
├── service/        # 业务逻辑层（接口 + impl）
├── manager/        # 通用能力/多表组合/外部调用封装（可选）
├── mapper/         # MyBatis-Plus Mapper
├── entity/         # 数据库实体（DO）
├── dto/            # 传输对象（Request/Response/VO）
├── converter/      # DO ↔ DTO 转换（MapStruct）
└── config/         # 模块级配置
```

调用链：`Controller → Service → Manager → Mapper`，禁止跨层调用与循环依赖。

### 4.3 核心技术架构要点

- **鉴权**：网关统一校验 JWT，解析出 userId 透传到下游（请求头 `X-User-Id`）；biz 内 `common-security` 提供 `UserContext` 获取当前用户。
- **Token 机制**：Access Token（2h）+ Refresh Token（30d），Redis 维护 Token 黑名单支持登出。
- **异步解析**：文件上传成功后，module-file 发送 RocketMQ 消息（`FILE_UPLOADED`），Python 解析服务消费；解析完成回发 `PARSE_DONE`，biz 更新笔记解析状态。
- **AI 透传**：module-ai 通过 gRPC 调用 Python search_service，问答结果以 SSE 形式经网关流式返回前端。
- **统一响应**：`Result<T>{ code, message, data }`，全局异常处理器统一封装。

### 4.4 第一阶段数据库表（MySQL 8.0）

完整初始化脚本见 `docs/phase1_database.sql`，以下为关键表结构概览：

```sql

CREATE DATABASE IF NOT EXISTS notegather
    DEFAULT CHARACTER SET utf8mb4
    DEFAULT COLLATE utf8mb4_unicode_ci;

-- 用户表（t_user）
-- 字段：id, username, password(BCrypt), nickname, avatar_url, status(1=正常/0=封禁), create_time, update_time, deleted

-- 特点：name_active 生成列 + 唯一索引实现同用户内知识库名称唯一约束

-- 笔记/文件夹节点表（t_note）
-- 字段：id, library_id, parent_id, user_id, node_type(FOLDER/NOTE), title, content(MEDIUMTEXT), note_type, sort_order, version, parse_status, create_time, update_time, deleted
-- 特点：支持多级树结构，parent_id 自引用外键 + ON DELETE CASCADE

-- 笔记版本表（t_note_version）
-- 字段：id, note_id, version, title, content(MEDIUMTEXT), created_by, create_time
-- 特点：不可变历史快照，用于编辑历史与回收站恢复

-- 特点：status_active 生成列实现防并发唯一约束

-- 聊天会话表（t_chat_session）
-- 字段：id, user_id, title, create_time, update_time, deleted

-- 聊天消息表（t_chat_message）
-- 字段：id, session_id, message_no, role(USER/ASSISTANT/SYSTEM), content(MEDIUMTEXT), status(STREAMING/COMPLETED/FAILED), error_msg, create_time, update_time, deleted

-- 聊天引用表（t_chat_citation）
-- 字段：id, message_id, note_id, citation_order, note_title, chunk_text, score, create_time
```

## 5. Python AI 端服务定位与架构设计

### 5.1 服务定位

Python AI 端在第一阶段承担两项核心职责：
1. **文档解析服务（parse_service）**：消费 RocketMQ 文件上传事件，完成文本抽取 → 分块 → 向量化 → 入库全链路。
2. **检索问答服务（search_service）**：承接来自 Java 的 gRPC 请求，完成向量检索 + ES 全文检索，组装上下文，调 LLM 流式回答。

### 5.5 第一阶段 gRPC 接口定义

```protobuf
// protos/ng_ai.proto
syntax = "proto3";
package ngai;

service ParseService {
  rpc GetParseStatus (ParseStatusRequest) returns (ParseStatusResponse);
}

service ChatService {
  rpc Chat (ChatRequest) returns (stream ChatChunk);
}

message ParseStatusRequest {
  string file_id = 1;
}

message ParseStatusResponse {
  string file_id = 1;
  string status  = 2;  // PENDING / PROCESSING / DONE / FAILED
  int32  chunk_count = 3;
}

message ChatRequest {
  string user_id  = 1;
  string question = 2;
  int32  top_k    = 3;
}

message ChatChunk {
  string content    = 1;  // token 片段
  bool   is_done    = 2;
  repeated Citation citations = 3;  // 仅在 is_done=true 时填充
}

message Citation {
  string note_id    = 1;
  string note_title = 2;
  string chunk_text = 3;
  float  score      = 4;
}
```

## 6. 模块间通信方案

### 6.1 通信架构图

```
Web / Android
     │
     │ HTTPS / SSE / WebSocket
     ▼
  ng-gateway (Spring Cloud Gateway)
     │ 鉴权 + 路由 + 限流
     ▼
  ng-biz (Spring Boot)
  ├── module-user    ─────────────────────── MySQL 8.0
  ├── module-knowledge ───────────────────── MySQL 8.0
  ├── module-file ─── RocketMQ(生产) ──────► parse_service
  │                                              │
  │                          RocketMQ(消费) ◄────┘
  │                          (PARSE_DONE)
  └── module-ai ──── gRPC ─────────────────► search_service
                     (ChatRequest/stream)        │
                                             Qdrant + ES
                     SSE 流式回调 ◄─────────────┘
```

### 6.2 通信协议选型说明

| 场景 | 协议 | 选型理由 |
|------|------|---------|
| 客户端 ↔ 网关 | REST（HTTPS）| 标准、可测试、跨平台 |
| 客户端 AI 问答 | SSE | 单向服务端推流，比 WebSocket 轻量，适合问答场景 |
| Java ↔ Python（查询/问答）| gRPC | 强类型、高性能、流式支持、IDL 先行保证契约 |
| 文件解析异步解耦 | RocketMQ | 事务消息保障消息不丢、事件驱动解耦 |
| 服务发现 | Nacos | 本阶段部署简单，可不用；生产环境接入 |

### 6.3 RocketMQ 消息设计

| Topic | Tag | 生产者 | 消费者 | 消息体 |
|-------|-----|--------|--------|--------|
| NG_FILE_TOPIC | FILE_UPLOADED | module-file | parse_service | `{fileId, userId, noteId, objectKey, fileType}` |
| NG_FILE_TOPIC | PARSE_DONE | parse_service | module-file | `{fileId, noteId, chunkCount, status}` |

事务消息：`FILE_UPLOADED` 使用 RocketMQ 事务消息，保证文件写入 MinIO 与消息发送的原子性（本地事务：更新 t_file.parse_status=PENDING）。

### 7.2 关键实现

- **SSE 问答**：使用 `EventSource` 连接 `GET /api/v1/ai/chat?question=xxx`，逐 token 渲染。
- **笔记树**：递归渲染知识库目录树，支持折叠展开，拖拽排序（Vue Draggable）。
- **文件上传**：分片上传大文件，上传后轮询 `GET /api/v1/files/{id}/status` 展示解析进度。

### 9.3 编码规范

**Java**

- 代码风格：Google Java Style Guide，IDEA 配置 `google-java-format` 插件
- 命名：类 `PascalCase`，方法/变量 `camelCase`，常量 `UPPER_SNAKE_CASE`，数据库字段 `lower_snake_case`
- 注释：公共接口必须写 JavaDoc，复杂逻辑写行内注释（中文可），禁止无意义注释
- 统一响应：所有 Controller 方法返回 `Result<T>`，统一通过 `Result.ok(data)` / `Result.fail(code, msg)` 构建
- 异常处理：自定义 `BusinessException`，全局 `@ControllerAdvice` 捕获并格式化响应
- 日志：使用 SLF4J + Logback，Controller 入口出口记录 DEBUG，业务异常记录 WARN，系统异常记录 ERROR；禁止 `System.out.println`
- 禁止：直接在 Controller 写业务逻辑；在 Service 直接操作 HTTP 响应

**Python**

- 代码风格：Black 格式化 + isort 排序导入，mypy 类型检查
- 命名：模块/函数/变量 `snake_case`，类 `PascalCase`
- 配置：所有敏感配置通过环境变量读取（`python-decouple` 或 `pydantic-settings`），禁止硬编码
- 日志：`loguru` 统一日志，gRPC 服务端记录每次请求的 user_id / latency

**通用**

- Git 分支策略：`main`（保护，不直接推）→ `develop` → `feature/xxx`，PR 合并到 develop
- Commit 规范：`feat: ...` / `fix: ...` / `refactor: ...` / `docs: ...` / `chore: ...`
- 不允许将配置文件（含密码、Key）提交到 Git，使用 `.env` 文件 + `.gitignore`

### 9.4 配置管理

- Java 多环境：`application.yml`（公共）+ `application-dev.yml`（开发）+ `application-prod.yml`（生产），通过 `spring.profiles.active` 切换
- Python 多环境：`.env.dev` / `.env.prod`，`python-decouple` 加载
- 敏感配置（数据库密码、LLM API Key）：本地 `.env` 文件，生产用 Nacos 配置中心或 K8s Secret

### 9.5 API 规范

- 路径格式：`/api/v1/{module}/{resource}[/{id}]`
- 版本管理：URL 携带版本号（`v1`），变更不兼容时升级版本
- 认证：Header `Authorization: Bearer <access_token>`
- 分页：统一 `page`（从 1 开始）+ `pageSize`，响应包含 `total`
- 时间：请求/响应统一 ISO 8601（`2026-07-13T08:00:00Z`），时区 UTC+8
