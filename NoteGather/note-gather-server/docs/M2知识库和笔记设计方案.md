# NoteGather M2 知识库和笔记设计方案

> 版本：v1.0  
> 阶段：Phase 1 / M2  
> 适用范围：`module-knowledge`、`module-file`、`common-security`  
> 最后更新：2026-07-14

## 1. 设计目标

M2 交付知识库、文件夹、笔记、目录树、移动排序、版本、回收站和文件上传绑定笔记的基础能力，形成后续文件解析与 RAG 引用跳转的内容底座。

第一阶段只实现个人空间内的资源管理，不实现协作权限、双向链接、标签系统、全文搜索、离线同步和 CRDT 实时编辑。所有接口都必须从 `UserContext` 获取当前用户 ID，并在 SQL 查询条件中显式带上 `user_id`，不能只依赖前端传入的 `libraryId`、`noteId` 或 `parentId`。

## 2. 当前依据与边界

现有数据库已经具备 M2 需要的四张核心表：

- `t_library`：知识库根容器，按 `user_id` 隔离。
- `t_note`：文件夹与笔记共表，`node_type=FOLDER|NOTE`，支持树结构。
- `t_note_version`：笔记版本快照。
- `t_file`：文件对象记录，可通过 `library_id` 与 `note_id` 绑定知识库和笔记。

当前 `module-knowledge` 只有包结构，需要补齐 Entity、Mapper、DTO、Service、Controller。`module-file` 后续上传文件时需要调用 `module-knowledge` 的内部能力，在目标知识库下自动创建一个笔记节点，并把 `t_file.note_id` 指向该笔记。

## 3. 总体架构

```text
ng-gateway
  -> 注入 X-User-Id
  -> ng-biz / module-knowledge
      -> UserContext 获取当前用户
      -> KnowledgeController 参数校验和 HTTP 响应
      -> LibraryService / NoteService 编排业务规则
      -> Mapper 访问 t_library / t_note / t_note_version
      -> module-file 上传完成后调用 NoteService 创建文件笔记
```

M2 不新增独立微服务，也不引入通用树引擎。树结构规则放在 `NoteService` 内聚实现，后续如果双向链接、协作权限、批量导入带来第二个明确场景，再考虑拆出专门的 TreePolicy 或 DomainService。

## 4. 数据模型约定

### 4.1 知识库 `t_library`

字段含义：

| 字段 | 约定 |
|---|---|
| `user_id` | 当前登录用户 ID，所有查询、更新、删除必须携带 |
| `name` | 同一用户活跃知识库范围内唯一 |
| `type` | `RESOURCE`、`SCRATCH`、`NOTE`、`DIARY` |
| `sort_order` | 知识库排序，非负整数 |
| `deleted` | `0` 未删除，`1` 已删除 |

知识库删除采用软删除。删除知识库时，第一阶段建议同步软删除其下所有未删除的 `t_note` 节点；`t_file` 不做物理删除，后续文件模块可基于 `library_id` / `note_id` 判断是否展示。

### 4.2 笔记节点 `t_note`

字段含义：

| 字段 | 约定 |
|---|---|
| `library_id` | 所属知识库，必须属于当前用户 |
| `parent_id` | 父文件夹 ID，`NULL` 表示直属知识库根 |
| `user_id` | 当前用户 ID，查询必须携带 |
| `node_type` | `FOLDER` 或 `NOTE` |
| `title` | 文件夹名或笔记标题 |
| `content` | 文件夹固定为空串，笔记保存编辑器正文 |
| `note_type` | 仅笔记有值：`NOTE`、`SCRATCH`、`TABLE` |
| `sort_order` | 同级排序 |
| `version` | 当前笔记版本号，从 1 开始 |
| `parse_status` | `NONE`、`PENDING`、`PROCESSING`、`DONE`、`FAILED` |
| `deleted` | `0` 未删除，`1` 回收站 |

`FOLDER` 节点不允许写入正文，也不允许有 `note_type`。`NOTE` 节点必须有 `note_type`，没有传时默认 `NOTE`。

### 4.3 笔记版本 `t_note_version`

每次更新笔记正文前，先把旧版本快照写入 `t_note_version`，再更新 `t_note` 的 `title`、`content` 和 `version + 1`。恢复版本时，同样先保存当前版本快照，再把目标快照内容写回 `t_note`，并生成新的当前版本号，避免“恢复”动作覆盖历史。

版本只针对 `node_type=NOTE` 生效，文件夹没有版本。

### 4.4 文件绑定规则

第一阶段明确规则：上传文件时，必须选择目标知识库，可选目标父文件夹；上传成功后 `module-file` 自动在目标位置创建一个笔记节点。

```text
上传文件
  -> 校验 libraryId / parentId 属于当前用户
  -> MinIO 保存对象
  -> t_file 先写入基础文件记录
  -> module-knowledge 创建文件笔记节点
  -> t_file.note_id 绑定新笔记
  -> 发布 FILE_UPLOADED 事件
```

文件自动创建的笔记：

- `node_type=NOTE`
- `note_type=NOTE`
- `title` 默认使用原始文件名去掉扩展名；如果重名，第一阶段允许重名，由前端靠 ID 区分。
- `content` 初始可以写入一个轻量占位，例如文件名、大小和解析状态；解析完成后由后续解析结果更新或追加。
- `parse_status=PENDING`

这样 `t_file.note_id`、检索 chunk metadata、`t_chat_citation.note_id` 和“点击引用跳回原笔记”能形成闭环。

## 5. 资源隔离原则

所有读写必须遵守以下规则：

1. `libraryId` 查询必须使用 `id = libraryId AND user_id = currentUserId AND deleted = 0`。
2. `noteId` 查询必须使用 `id = noteId AND user_id = currentUserId`，普通业务默认加 `deleted = 0`。
3. `parentId` 非空时，必须校验父节点存在、属于当前用户、属于同一知识库、`node_type=FOLDER`、未删除。
4. 更新、移动、排序、删除、恢复都不能只按主键更新，必须带 `user_id` 条件。
5. 返回目录树时只返回当前用户当前知识库下的数据。

建议在 `LibraryService` / `NoteService` 提供私有方法：

- `requireLibrary(userId, libraryId, includeDeleted)`
- `requireNote(userId, noteId, includeDeleted)`
- `requireFolder(userId, libraryId, parentId)`

不要让 Controller 直接访问 Mapper，也不要把资源归属校验留给前端。

## 6. 树结构规则

### 6.1 最大五层

产品约定“知识库 -> 文件夹 -> 笔记，最多五层”。M2 建议按节点层级计算：

```text
知识库根 = 0
直属知识库的节点 = 1
最多允许节点层级 = 5
```

也就是说，一个节点从自身向上追溯到根，节点数量不能超过 5。创建和移动时都必须校验。

### 6.2 父子关系校验

创建节点：

- `parentId=NULL`：节点直属知识库。
- `parentId!=NULL`：父节点必须是同一知识库下的未删除文件夹。

移动节点：

- 目标父节点可以为 `NULL`。
- 目标父节点不能是自己。
- 目标父节点不能是自己的任意后代。
- 移动后的整棵子树最大深度不能超过 5。
- 节点和目标父节点必须属于同一用户。

### 6.3 循环引用校验

移动节点时，从目标父节点开始向上追溯 `parent_id`：

```text
targetParent -> parent -> parent ...
```

如果追溯过程中遇到当前移动节点 ID，说明会形成循环，必须拒绝。因为最大深度只有 5，第一阶段不需要复杂递归 SQL，Service 内循环查询即可，代码更直观。

### 6.4 排序规则

排序只在同一知识库、同一父节点下生效。

建议提供两种接口：

- 移动节点时附带 `targetParentId` 和 `sortOrder`，完成跨父节点移动。
- 批量排序接口接收同级节点 ID 列表，按列表顺序写入 `sort_order`。

批量排序必须校验所有节点：

- 都属于当前用户。
- 都属于同一知识库。
- 都属于同一个 `parent_id`。
- 都未删除。

## 7. API 契约

统一前缀：`/api/v1`。

### 7.1 知识库

```http
POST   /api/v1/libraries
GET    /api/v1/libraries
GET    /api/v1/libraries/{libraryId}
PUT    /api/v1/libraries/{libraryId}
DELETE /api/v1/libraries/{libraryId}
POST   /api/v1/libraries/{libraryId}/restore
```

创建请求：

```json
{
  "name": "资料库",
  "type": "RESOURCE",
  "description": "论文、PDF、资料归档",
  "sortOrder": 0
}
```

查询列表默认只返回未删除知识库。回收站接口单独处理，避免普通列表混入删除数据。

### 7.2 目录树

```http
GET /api/v1/libraries/{libraryId}/tree
```

返回结构：

```json
{
  "libraryId": "1",
  "nodes": [
    {
      "noteId": "10",
      "parentId": null,
      "nodeType": "FOLDER",
      "title": "论文",
      "sortOrder": 0,
      "children": []
    }
  ]
}
```

树构建在 Service 内完成：一次查出知识库下未删除节点，按 `parent_id` 分组，按 `sort_order,id` 排序后组装。最大五层数据量可控，不需要递归 SQL。

### 7.3 文件夹和笔记

```http
POST   /api/v1/notes
GET    /api/v1/notes/{noteId}
PUT    /api/v1/notes/{noteId}
DELETE /api/v1/notes/{noteId}
POST   /api/v1/notes/{noteId}/restore
```

创建文件夹：

```json
{
  "libraryId": "1",
  "parentId": null,
  "nodeType": "FOLDER",
  "title": "论文",
  "sortOrder": 0
}
```

创建笔记：

```json
{
  "libraryId": "1",
  "parentId": "10",
  "nodeType": "NOTE",
  "noteType": "NOTE",
  "title": "Transformer 阅读笔记",
  "content": "# Transformer",
  "sortOrder": 0
}
```

更新笔记时：

- 文件夹只能更新 `title` 和 `sortOrder`。
- 笔记可更新 `title`、`content`、`noteType`。
- 只有 `content` 或 `title` 发生变化时保存版本快照。
- 更新笔记后 `parse_status` 建议置为 `PENDING`，供后续解析/索引增量任务消费；如果 M2 暂不接 MQ，可以先只更新数据库状态。

### 7.4 移动和排序

```http
POST /api/v1/notes/{noteId}/move
POST /api/v1/notes/reorder
```

移动请求：

```json
{
  "targetParentId": "20",
  "sortOrder": 3
}
```

排序请求：

```json
{
  "libraryId": "1",
  "parentId": "20",
  "orderedNoteIds": ["21", "22", "23"]
}
```

### 7.5 版本

```http
GET  /api/v1/notes/{noteId}/versions
POST /api/v1/notes/{noteId}/versions/{version}/restore
```

版本列表只返回当前用户有权访问的笔记版本。恢复指定版本必须校验 `note_id` 对应笔记属于当前用户，不能只按 `note_version.id` 或 `version` 查询。

### 7.6 回收站

```http
GET    /api/v1/recycle-bin/libraries
GET    /api/v1/recycle-bin/notes?libraryId=1
POST   /api/v1/notes/{noteId}/restore
DELETE /api/v1/recycle-bin/notes/{noteId}
```

第一阶段建议：

- 普通删除是软删除，进入回收站。
- 恢复时只恢复当前节点，不自动恢复已删除父节点；如果父节点已删除或不存在，恢复到知识库根。
- 删除文件夹时，其子树一起软删除。
- 彻底删除可以先不做物理删除，M2 可把接口保留为“二次软删除无操作”会让语义变怪；建议第一阶段先不开放彻底删除，除非确认需要。

## 8. 事务策略

需要事务的方法：

- 创建知识库。
- 删除知识库及其节点。
- 创建笔记。
- 更新笔记并保存版本。
- 恢复版本。
- 移动节点。
- 批量排序。
- 删除/恢复节点子树。
- 上传文件后自动创建笔记并绑定 `t_file.note_id`，这一块在 M4 实现时需要 `module-file` 和 `module-knowledge` 在同一个 `ng-biz` 进程内以本地事务协作。

不建议把 MinIO、RocketMQ、MySQL 强行塞进一个普通数据库事务。M4 文件上传可以采用“先落对象和文件记录，再创建笔记，再发消息”的事务消息或补偿策略；M2 只定义绑定规则。

## 10. 与 module-file 的协作

M2 先在 `module-knowledge` 暴露一个进程内 Service 方法：

```text
createFileNote(userId, libraryId, parentId, fileName, initialContent)
```

M4 文件上传实现时，`module-file` 直接依赖 `module-knowledge` 不合适，因为当前 Maven 模块同级，容易形成模块循环。推荐抽一个窄接口放到 `common-api`：

```text
KnowledgeFacade.createFileNote(...)
```

由 `module-knowledge` 实现，`module-file` 通过 Dubbo 或同进程 Spring Bean 使用。第一阶段如果都聚合在 `ng-biz-app`，优先用 Spring Bean，避免为同进程调用引入网络 RPC；但接口定义仍放 `common-api`，保持依赖方向干净。

## 11. 错误码与异常

建议补充或复用以下错误：

| 场景 | HTTP | 说明 |
|---|---:|---|
| 知识库不存在 | 404 | 不存在、已删除或不属于当前用户时统一返回 |
| 笔记不存在 | 404 | 不存在、已删除或不属于当前用户时统一返回 |
| 父节点非法 | 400 | 父节点不是文件夹、跨知识库、已删除 |
| 超过最大层级 | 400 | 创建或移动后超过 5 层 |
| 循环移动 | 400 | 移动到自身或后代下 |
| 知识库名称冲突 | 409 | 同用户活跃知识库重名 |
| 节点类型非法 | 400 | 文件夹带正文、笔记缺少 noteType 等 |
| 版本不存在 | 404 | 指定版本不存在或不属于当前用户笔记 |

错误返回继续使用 `Result<T>` 和 `GlobalExceptionHandler`。

## 12. 实现顺序

1. 补充 `Library`、`Note`、`NoteVersion` Entity、Mapper、枚举和 DTO。

### 分层结构校正（2026-07-15）

`module-knowledge` 与 `module-file` 运行在 `ng-biz` 服务内，但不是前端可调用的独立服务。跨模块协作通过窄的 `KnowledgeFacade` 完成，文件模块不会直接操作笔记表。

```text
module-knowledge
├── domain/model, domain/repository
├── application/service
├── infrastructure/persistence/*/mapper
├── infrastructure/persistence/*/repository
└── adapter/rpc

module-file
├── domain/model, domain/enums, domain/repository
├── application/config, application/dto, application/port, application/service
├── infrastructure/config, infrastructure/persistence, infrastructure/storage, infrastructure/messaging
└── adapter/web, adapter/messaging
```

应用服务通过 Repository Port 完成资源归属查询、写入及解析状态更新；MinIO 与 RocketMQ 分别实现 `FileStorage`、`FileUploadedEventPublisher` 应用端口。这样即使实现技术替换，知识库、笔记和文件的用户隔离规则不需要进入 Controller 或基础设施代码。
2. 实现 `LibraryService`：创建、列表、详情、更新、软删除、恢复。
3. 实现 `NoteService` 基础 CRUD：创建文件夹、创建笔记、详情、更新、删除。
4. 实现资源归属校验：所有 `libraryId` / `noteId` / `parentId` 都基于当前 `userId` 校验。
5. 实现目录树：一次查询、内存组树、排序。
6. 实现移动节点：父子关系、循环引用、最大五层、子树深度校验。
7. 实现同级批量排序。
8. 实现笔记版本保存、版本列表、版本恢复。
9. 实现回收站列表和恢复。
10. 定义 `common-api` 中给文件模块使用的 `createFileNote` 窄接口，为 M4 上传绑定笔记做准备。
11. 补单元测试和接口集成测试。

## 13. 测试与验收

### 13.1 单元测试

- 创建直属节点层级为 1，最多允许到 5。
- 第 6 层创建失败。
- 移动节点到自身失败。
- 移动节点到后代失败。
- 跨知识库移动失败。
- 父节点是笔记时创建/移动失败。
- 更新笔记会保存旧版本快照。
- 恢复版本会生成新的当前版本。
- 回收站恢复时，父节点不存在或已删除则恢复到根。

### 13.2 接口集成测试

```text
1. 创建知识库，列表可见。
2. 创建文件夹和笔记，目录树结构正确。
3. 更新笔记后，版本列表出现旧版本。
4. 恢复旧版本，当前内容回到历史内容，同时版本号继续递增。
5. 移动节点到另一个文件夹，目录树位置正确。
6. 批量排序后，同级节点顺序正确。
7. 删除笔记后，普通树不可见，回收站可见。
8. 恢复笔记后，普通树可见，回收站不可见。
9. 用户 A 不能通过用户 B 的 libraryId 或 noteId 查询、更新、删除资源。
10. 上传文件创建笔记的内部接口能返回 noteId，供 t_file.note_id 绑定。
```

### 13.3 交付验收

M2 完成的最低标准：

- 知识库 CRUD 可用。
- 文件夹和笔记 CRUD 可用。
- 目录树展示正确。
- 移动、排序、五层限制、循环引用校验可验证。
- 笔记版本保存和恢复可用。
- 软删除、回收站、恢复可用。
- 所有接口基于当前用户隔离资源。
- 文件上传自动创建笔记的接口契约明确，能支撑 M4 与 RAG 引用闭环。

## 14. 方案自检

- **Hard-coding**：最大层级、节点类型、知识库类型、解析状态应通过常量或 enum 管理；接口路径集中在 Controller，不散落到业务逻辑。
- **Coupling/Cohesion**：知识库和树规则留在 `module-knowledge`，文件上传只通过窄接口请求创建文件笔记，不直接操作 `t_note`。
- **Type Safety**：请求 DTO 使用 Bean Validation，类型字段使用 enum，资源 ID 在 Service 层统一转换和校验，所有 Mapper 查询显式带 `user_id`。
