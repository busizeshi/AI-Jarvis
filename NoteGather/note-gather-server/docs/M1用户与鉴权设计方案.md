# NoteGather M1 用户与鉴权设计方案

> 版本：v1.0  
> 阶段：Phase 1 / M1  
> 适用范围：`ng-admin`、`common-security`、`common-redis`、`ng-gateway`
> 最后更新：2026-07-14

## 1. 设计目标

M1 交付用户名密码注册登录、Access Token / Refresh Token、登出、当前用户信息和下游用户上下文能力，满足以下闭环：

```text
注册 → 登录 → 获取 Token → 访问受保护接口
Access Token 过期 → Refresh Token 换新
登出 → 原 Token 无法继续访问
```

本阶段不实现 OAuth、微信/Google 登录、RBAC、设备管理和多端登录管理。认证入口统一由网关承载，业务服务只接收网关注入的内部用户上下文。

## 2. 现状与调整

当前工程已经具备：

- `JwtUtils`：JWT 生成、解析和 Token 前缀处理。
- `JwtProperties`：JWT 密钥、Access Token 和 Refresh Token 有效期配置。
- `RedisService`：String、Hash、Set 和过期时间操作。
- `JwtAuthFilter`：网关 Token 提取、黑名单查询和用户信息透传。
- `UserContext`：保存当前请求的 `LoginUser`。

当前需要补齐：

- `ng-admin` 内用户域的 domain、application、infrastructure、adapter、bootstrap 分层。
- Token 类型校验，禁止 Refresh Token 访问普通业务接口。
- Refresh Token 的 Redis 会话、轮换和重放检测。
- biz 服务从 `X-User-Id` / `X-Username` 建立并清理 `UserContext` 的 Servlet Filter。
- 注册、登录、刷新、登出、当前用户信息的测试。

## 3. 总体架构

```text
客户端
  │ Authorization: Bearer <access-token>
  ▼
ng-gateway
  ├─ 白名单接口直接放行
  ├─ 校验 JWT 签名、有效期、type=access
  ├─ 查询 Redis 黑名单
  ├─ 删除客户端伪造的 X-User-Id / X-Username
  └─ 注入可信的 X-User-Id / X-Username
       │
       ▼
ng-admin
  ├─ UserContextFilter：读取内部请求头并设置 UserContext
  ├─ UserController：参数校验和 HTTP 响应
  ├─ UserService：注册、登录、刷新、登出业务编排
  ├─ JwtUtils：生成和验证 Token
  ├─ RedisService：Refresh Token 会话和黑名单
  └─ UserMapper：访问 t_user
```

网关是唯一对外暴露的认证边界，`ng-admin` 与 `ng-biz` 都不应直接暴露到公网。`X-User-Id` 只在网关到内部服务的网络中使用；网关必须覆盖客户端同名请求头，不能直接沿用客户端传入的值。

本方案不额外引入认证中心、Session 服务或通用策略框架，避免在只有一个登录场景时增加不必要的抽象层。

## 4. Token 设计

### 4.1 Token 类型

| Token | 用途 | 默认有效期 | 是否可访问业务接口 | 是否存 Redis |
|---|---|---:|:---:|:---:|
| Access Token | 访问网关和业务 API | 2 小时 | 是 | 否 |
| Refresh Token | 换取新 Token | 30 天 | 否 | 是 |

Access Token 和 Refresh Token 使用同一个 JWT 签名密钥，但通过 `type` Claim 严格区分。所有时间值必须来自 `JwtProperties`，业务代码不得自行写入固定毫秒数。

### 4.2 JWT Claims

```json
{
  "sub": "用户ID",
  "uid": "用户ID",
  "uname": "用户名",
  "nick": "昵称",
  "jti": "随机Token ID",
  "type": "access | refresh",
  "iat": 0,
  "exp": 0
}
```

要求：

- `jti` 使用密码学安全的随机值，每个 Token 唯一。
- `uid`、`uname` 只用于构造上下文，不作为用户数据的最终来源。
- `type=access` 才能通过网关普通鉴权。
- `type=refresh` 只能用于刷新接口。
- `parseToken` 不应只返回用户信息而丢弃 Token 类型和 `jti`，建议返回包含 Claims 的内部 Token 结果对象，或提供明确的 `parseAccessToken` / `parseRefreshToken` 方法。

### 4.3 Redis Key

统一定义在 `CommonConstants` 中：

| Key | Value | TTL | 用途 |
|---|---|---:|---|
| `ng:token:refresh:{jti}` | `userId` 或 JSON 会话信息 | Refresh Token 剩余有效期 | 判断 Refresh Token 是否仍是当前有效会话 |
| `ng:token:blacklist:{jti}` | `1` | 对应 Token 剩余有效期 | 登出或主动失效后的 Token 拦截 |

不建议把完整 JWT 放入 Redis Key。使用 `jti` 可以降低日志、监控和 Redis 运维中泄漏完整 Token 的风险，也便于后续实现 Token 轮换。

### 4.4 Refresh Token 轮换

刷新流程必须是一次性消费旧 Refresh Token：

```text
客户端提交 refreshToken
  ↓
验证签名、exp、type=refresh、jti
  ↓
读取 ng:token:refresh:{jti}
  ├─ 不存在：拒绝，视为过期或重放
  └─ 存在：校验 userId 和账号状态
  ↓
删除旧 jti 会话并加入旧 Token 黑名单
  ↓
生成新的 Access Token + Refresh Token
  ↓
写入新的 Refresh Token 会话
  ↓
返回新的 Token 对
```

并发刷新时必须避免两个请求同时消费同一个 Refresh Token。第一阶段可以使用 Redis `SETNX` 或分布式锁，以 `jti` 作为锁粒度；如果旧会话删除失败，刷新操作必须失败，不能继续返回新 Token。

## 5. API 契约

统一前缀：`/api/v1/user`。普通业务接口使用 `Authorization: Bearer <access-token>`。

### 5.1 注册

```http
POST /api/v1/user/register
Content-Type: application/json
```

请求：

```json
{
  "username": "alice",
  "password": "******",
  "nickname": "Alice"
}
```

规则：

- `username` 去除首尾空白后校验，长度 3-64。
- 用户名在活跃用户范围内唯一。
- `password` 不保存明文，只保存 BCrypt 哈希。
- 注册成功不自动签发 Token，登录流程单独负责签发。
- 返回用户基本信息，不返回密码和内部安全字段。

### 5.2 登录

```http
POST /api/v1/user/login
Content-Type: application/json
```

请求：

```json
{
  "username": "alice",
  "password": "******"
}
```

响应数据：

```json
{
  "accessToken": "...",
  "refreshToken": "...",
  "tokenType": "Bearer",
  "accessTokenExpiresIn": 7200,
  "refreshTokenExpiresIn": 2592000,
  "user": {
    "userId": "1",
    "username": "alice",
    "nickname": "Alice"
  }
}
```

登录前必须检查用户存在、密码匹配、账号状态正常。账号不存在、密码错误和账号被禁用应使用明确的业务错误码；对外是否合并用户名不存在和密码错误的提示，可在 Controller 层统一为“用户名或密码错误”，避免暴露账号存在性。

### 5.3 刷新 Token

```http
POST /api/v1/user/refresh-token
Content-Type: application/json
```

请求：

```json
{
  "refreshToken": "..."
}
```

该接口只接受 Refresh Token。Access Token、格式错误的 Token、已过期 Token、已轮换 Token 和已登出 Token 都必须拒绝。

### 5.4 登出

```http
POST /api/v1/user/logout
Authorization: Bearer <access-token>
```

登出动作：

- 将当前 Access Token 的 `jti` 加入黑名单，TTL 设置为其剩余有效期。
- 删除当前用户的 Refresh Token 会话，或将对应 Refresh Token `jti` 加入黑名单。
- 即使 Redis 中不存在对应刷新会话，登出接口也应保持幂等。

如果客户端同时提交 Refresh Token，可在请求体中携带；服务端必须先验证其签名和类型，再执行失效操作，不能盲目把未经验证的字符串写入 Redis。

### 5.5 当前用户信息

```http
GET /api/v1/user/me
Authorization: Bearer <access-token>
```

接口从 `UserContext` 获取用户 ID，再从数据库查询最新用户信息。不要直接使用 JWT 中的昵称作为最终返回值，避免用户修改资料后 Token 内信息过期。

## 6. Java 模块职责

### 6.1 `common-security`

建议调整和新增：

```text
common-security/
└── src/main/java/com/notegather/common/security/
    ├── config/JwtProperties.java
    ├── context/UserContext.java
    ├── filter/UserContextFilter.java
    ├── model/LoginUser.java
    ├── model/TokenPayload.java
    └── util/JwtUtils.java
```

职责边界：

- `JwtUtils`：签发、解析、校验 Token 签名、过期时间和类型。
- `UserContext`：只负责当前请求用户上下文的读写和清理。
- `UserContextFilter`：只负责从内部请求头构造上下文，不查询数据库、不处理登录业务。
- `JwtProperties`：集中管理密钥、有效期、Header 名称和前缀。

`UserContext` 应使用普通 `ThreadLocal`，并在 Filter 的 `finally` 中清理。`InheritableThreadLocal` 在 Servlet 线程池中可能造成上下文继承和串请求风险，不适合作为这里的默认实现。

### 6.2 `ng-admin` 用户域分层

```text
ng-admin/src/main/java/com/notegather/admin/
└── src/main/java/com/notegather/biz/user/
    ├── controller/UserController.java
    ├── converter/UserConverter.java
    ├── dto/
    │   ├── RegisterRequest.java
    │   ├── LoginRequest.java
    │   ├── RefreshTokenRequest.java
    │   ├── TokenResponse.java
    │   └── UserResponse.java
    ├── entity/User.java
    ├── mapper/UserMapper.java
    ├── service/UserService.java
    ├── service/impl/UserServiceImpl.java
    └── config/UserSecurityConfig.java
```

调用方向固定为：

```text
Controller → Service → Mapper / RedisService / JwtUtils
```

Controller 不直接访问 Mapper、Redis 或 JWT 细节。第一阶段不新增 UserManager、AuthFacade 等仅转发调用的包装层。

#### 结构校正（2026-07-15）

上述历史目录示例已由下列实际结构替代，`ng-admin` 不再保留或依赖 `module-user`：

```text
ng-admin/src/main/java/com/notegather/admin/
├── domain/user/model/User.java
├── domain/user/repository/UserRepository.java
├── application/user/dto/
├── application/user/assembler/UserAssembler.java
├── application/user/service/UserService.java
├── infrastructure/persistence/user/mapper/UserMapper.java
├── infrastructure/persistence/user/repository/MybatisUserRepository.java
├── adapter/web/user/UserController.java
├── adapter/rpc/user/UserFacadeImpl.java
└── bootstrap/config/UserSecurityConfig.java
```

调用方向为 `adapter -> application -> domain`，`MybatisUserRepository` 在基础设施层实现 `UserRepository`。应用服务不直接依赖 `UserMapper`；Redis 和 JWT 仍由应用服务通过已有的公共安全、缓存能力完成鉴权用例编排。

### 6.3 BCrypt

`ng-admin` 显式依赖 `spring-security-crypto`，由 bootstrap 配置提供单例 `PasswordEncoder`：

```java
new BCryptPasswordEncoder()
```

BCrypt 的 cost 使用框架默认值或配置值，不在注册逻辑中硬编码。密码校验统一调用 `passwordEncoder.matches(rawPassword, encodedPassword)`。

### 6.4 用户上下文过滤器

过滤器行为：

1. 判断请求路径是否属于登录、注册、刷新 Token、健康检查等白名单。
2. 白名单请求不要求 `X-User-Id`。
3. 其他请求读取 `X-User-Id` 和 `X-Username`。
4. 校验 `X-User-Id` 非空且可解析为 Long。
5. 构造 `LoginUser` 并写入 `UserContext`。
6. 调用后续过滤器链。
7. 在 `finally` 中清理 `UserContext`。

过滤器不再解析客户端 Authorization，因为网关已经完成 Access Token 校验。为了防止 biz 被绕过，生产部署必须限制 biz 端口的网络访问；后续可以增加内部服务签名，但不在 M1 引入第二套鉴权协议。

## 7. 网关鉴权调整

`JwtAuthFilter` 需要调整为以下规则：

- 普通受保护接口只接受 `type=access`。
- `/api/v1/user/refresh-token` 不进入普通 Access Token 鉴权，由 Controller 校验请求体中的 Refresh Token。
- 透传用户信息前先删除客户端传入的 `X-User-Id`、`X-Username` 和内部认证头。
- Redis 黑名单查询异常时默认失败关闭，不能在生产环境仅凭 JWT 继续放行。
- 网关返回 401 时统一使用 `Result` 格式。

建议将鉴权白名单集中配置，不在 Filter 中写死路径。当前配置已有白名单属性，可继续沿用。

## 8. 数据库与状态约定

使用现有 `t_user` 表，不新增认证表。需要统一以下含义：

| 字段 | 约定 |
|---|---|
| `status=1` | 账号正常，可登录 |
| `status=0` | 账号禁用，不可登录 |
| `deleted=0` | 活跃用户 |
| `deleted=1` | 逻辑删除用户 |
| `password` | BCrypt 哈希，不允许返回 API |

现有 `UserInfoDTO` 中关于 `status` 的注释必须与数据库脚本统一，避免“1 正常”和“1 禁用”在不同模块中含义相反。

注册事务只负责插入用户。登录、刷新和登出涉及 Redis，不应把 Redis 操作伪装成 MySQL 事务；失败时通过明确异常和幂等策略处理。

## 9. 异常与错误码

建议补充或统一以下业务错误：

| 错误场景 | HTTP | 说明 |
|---|---:|---|
| 参数校验失败 | 400 | 用户名、密码、Token 为空或格式不正确 |
| 未登录 | 401 | 缺少 Access Token 或内部用户头 |
| Token 过期 | 401 | Access Token 或 Refresh Token 已过期 |
| Token 类型错误 | 401 | Refresh Token 用于业务接口，或 Access Token 用于刷新 |
| Token 已失效 | 401 | 黑名单、已轮换或已登出 |
| 用户名已存在 | 409 | 活跃用户名冲突 |
| 用户名或密码错误 | 401 | 登录凭据不匹配 |
| 账号已禁用 | 403 | 用户状态不允许登录或访问 |
| Redis 不可用 | 503 | 无法安全完成黑名单或刷新会话校验 |

所有异常都通过 `GlobalExceptionHandler` 转换为统一 `Result`，不得把 JWT 原始异常、数据库异常或 Redis 连接信息直接返回给客户端。

## 10. 实现顺序

1. 修正 `UserContext` 为普通 `ThreadLocal`，增加 `UserContextFilter`。
2. 扩展 `JwtUtils`：增加 `jti`、Token 类型解析和 Access/Refresh 专用校验方法。
3. 补充 Token、Redis Key 和错误码常量。
4. 实现 `User` Entity、Mapper、DTO 和 `PasswordEncoder` 配置。
5. 实现注册和当前用户信息接口。
6. 实现登录和 Refresh Token Redis 会话。
7. 实现 Refresh Token 轮换、重放检测和登出。
8. 调整网关鉴权顺序、白名单和 Redis 异常策略。
9. 编写单元测试和接口集成测试。
10. 使用 Postman、curl 或前端客户端完成验收。

## 11. 测试与验收标准

### 11.1 单元测试

- BCrypt 密码加密后不能等于明文，正确密码可以匹配，错误密码不能匹配。
- Access Token 的 `type` 必须为 `access`。
- Refresh Token 的 `type` 必须为 `refresh`。
- 过期 Token、错误签名 Token、缺少 `jti` 的 Token 都被拒绝。
- Token 轮换后旧 Refresh Token 不能再次使用。
- `UserContext` 在正常请求和异常请求结束后都被清理。

### 11.2 接口集成测试

```text
1. 注册新用户，数据库中保存 BCrypt 哈希。
2. 使用相同用户名注册，返回 409。
3. 使用正确密码登录，返回 Access Token 和 Refresh Token。
4. 使用 Access Token 访问 /api/v1/user/me，返回最新用户信息。
5. 使用 Refresh Token 刷新，返回新的 Token 对。
6. 再次使用旧 Refresh Token 刷新，返回 401。
7. 登出后访问受保护接口，返回 401。
8. 使用 Refresh Token 访问 /api/v1/user/me，返回 401。
9. 禁用用户后，旧 Access Token 访问需要按策略拒绝；至少下次刷新必须失败。
10. Redis 不可用时，黑名单和刷新校验接口返回 503 或 401，不得默认放行。
```

### 11.3 交付验收

M1 完成的最低标准：

- 注册、登录、刷新、登出、当前用户信息接口可用。
- 网关可以阻断无效 Access Token 和 Refresh Token 类型混用。
- biz 服务可以从 `X-User-Id` 建立 `UserContext`，请求结束后不会污染线程池。
- Refresh Token 轮换和重放检测可验证。
- Redis、MySQL 不可用时有明确失败行为。
- 不返回密码、JWT 原始异常、数据库连接信息和 Redis 连接信息。

## 12. 方案自检

- **硬编码**：Token 有效期、Header、前缀、Redis Key 前缀和白名单均由配置或常量统一管理。
- **低耦合**：网关负责边界鉴权，`UserContextFilter` 负责上下文，`UserService` 负责用户业务，三者不互相承载职责。
- **类型安全**：请求 DTO 使用 Bean Validation，Token 通过 `type` 和 `jti` 校验，数据库状态和错误码统一定义。
